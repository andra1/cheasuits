Section 1: Database Schema

Two new tables, columns removed from properties, denormalized market value stays

New Table: valuations

valuations
Column Type Notes
idINTEGER PKautoincrement
document_numberTEXT FK→ properties.document_number
sourceTEXT NOT NULL"zillow" | "redfin" | "comps"
estimateREAL NOT NULLdollar amount
source_urlTEXTZillow/Redfin property page URL (null for comps)
confidenceTEXT"high" | "medium" | "low"
comp_countINTEGERonly for source="comps"
valued_atTEXT NOT NULLISO timestamp

UNIQUE(document_number, source) — one estimate per source per property, upserted on re-runs

New Table: property_comps

property_comps
Column Type Notes
idINTEGER PKautoincrement
document_numberTEXT FK→ properties.document_number (subject property)
comp_sale_idINTEGER FK→ comparable_sales.id
distance_milesREALhaversine distance
similarity_scoreREAL0-1 composite score (distance + recency + lot)
lot_size_ratioREALsubject_lot / comp_lot (clamped 0.5-2.0)
adjusted_priceREALsale_price * lot_size_ratio
matched_atTEXTISO timestamp

UNIQUE(document_number, comp_sale_id) — each comp linked once per subject property

New Table: comparable_sales — add id column

comparable_sales — modification

Current PK is (address, sale_date, sale_price). Add an INTEGER PRIMARY KEY id (autoincrement) so property_comps can reference individual rows cleanly. Keep the existing unique constraint as a dedup index.
